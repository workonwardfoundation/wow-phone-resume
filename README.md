# Call-Based Resume Backend

This project implements a backend server for the WorkOnward Resume Assistant using Fastify, WebSockets, and MongoDB. It is designed to handle Twilio voice calls and provide a conversational interface for resume creation using Azure OpenAI integration.

## Features

- **Twilio Integration:**  
  Handles incoming voice calls via Twilio webhooks.

- **Multi-Language Support:**  
  Supports multiple languages for a personalized user experience.

- **WebSocket Media Streaming:**  
  Establishes a WebSocket connection for streaming media (audio) data.

- **MongoDB Storage:**  
  Saves conversation logs and call transcripts to MongoDB with retry and reconnection mechanisms.

- **Structured Logging:**  
  Uses Pino for structured logging with different log levels (debug, info, error).

- **Centralized Error Handling:**  
  Implements error handling middleware for cleaner code and better debugging.

- **Swagger API Documentation (Optional):**  
  Optionally provides interactive API docs using Fastify Swagger.

## Setup

### 1. Install Dependencies

Clone the repository and install the required packages:

```bash
git clone https://github.com/yourusername/call-based-resume-backend.git
cd call-based-resume-backend
npm install
```

### 2. Install Extensions

Install following extensions:

- [Eslint](https://marketplace.visualstudio.com/items?itemName=dbaeumer.vscode-eslint)
- [Prettier](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode)

### 3. Environment Variables

Create a .env file in the project root with the following variables (adjust as needed):

```
PORT=80
MONGODB_URI=your_mongodb_connection_string
AZURE_OPENAI_ENDPOINT=your_azure_openai_endpoint
AZURE_OPENAI_API_KEY=your_azure_openai_api_key
AZURE_OPENAI_DEPLOYMENT_NAME=your_deployment_name
```

### 4. Running the Server

For development

```
npm run dev
```

For production

```
npm run build
npm run start
```

If you are using ngrok to expose your local server to the internet for Twilio, run:

```
ngrok http 80
```

### 5. Commit

This project uses [Husky](https://typicode.github.io/husky/) and [lint-staged](https://github.com/okonet/lint-staged) to ensure code quality before each commit.

When you run `git commit`, the following pre-commit checks will automatically run:

- `import-sort --write`: Automatically sorts your imports.
- `prettier --write`: Formats your code using Prettier.
- `eslint 'src/**/*.{ts,tsx}' --max-warnings=0`: Lints your TypeScript/TSX files. The commit will be blocked if there are any lint warnings or errors.

These steps are defined in the `lint-staged` configuration:

```json
"lint-staged": {
  "*.ts": [
    "import-sort --write",
    "prettier --write",
    "eslint 'src/**/*.{ts,tsx}' --max-warnings=0"
  ]
}
```

⚠️ Prettier Permission Error (EACCES)
`[FAILED] prettier --write [EACCES]`
It means that Prettier doesn’t have the necessary execute permissions.

To fix this, run the following command:
`chmod +x ./node_modules/.bin/prettier`

| Feature | AWS ECS Fargate | Google Cloud Run |
|---------|-----------------|------------------|
| **Setup Complexity** | Medium | Easy |

| **Scaling** | Manual + Auto-scaling | Automatic |

# Logging and Error Handling

## Logging

- The application uses **Pino** for logging.
- Logging is configured directly in `@/server` using Fastify’s built-in logger. Use `fastify.log.info`, `fastify.log.warn`, etc., after importing fastify from `@/server`.
- Logs include request details, error messages, and system events.

## Error Handling

- Centralized error handling middleware (`/src/middlewares/errorHandler.ts`) ensures that errors are logged and formatted consistently.
